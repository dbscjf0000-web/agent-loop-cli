# Plan: TDD 통합 — Sub-task 기반 4중 안전망

작성일: 2026-05-06
출처: Claude / Codex / Gemini 3자 토의

---

## 1. 배경

```
┌──────────────────────────────────────────────┐
│  현재 상태                                     │
│   • Phase 1 (5가지 quick wins) 구현 완료       │
│     236/239 → 239/239 통과                    │
│   • P phase는 plan.md에서 sub-task 분해 (자연어)│
│   • V phase는 rubric axis만 평가              │
│   • → P와 V가 직접 연결되지 않음               │
└──────────────────────────────────────────────┘
```

## 2. 핵심 발견

```
┌──────────────────────────────────────────────┐
│  "P가 sub-task 분해는 이미 함.                │
│   차이는 V가 그걸 직접 검증에 쓰느냐."          │
└──────────────────────────────────────────────┘
```

## 3. 제안 — 4단계 통합

### A. P 프롬프트 강화

```
┌────────────────────────────────────────────┐
│  현재 plan.md "구현 단계"                     │
│   1. parse 함수 작성                          │
│   2. core 알고리즘                            │
│   3. format                                  │
│  → 자연어 한 줄                               │
│                                              │
│  강화 후                                       │
│   ### subtask-1: parse                       │
│    - goal: 입력을 정수 리스트로               │
│    - acceptance:                             │
│        parse("1 2 3") == [1, 2, 3]          │
│        parse("") == []                       │
│    - test_hint: empty / 음수 / 공백 검증       │
└────────────────────────────────────────────┘
```

### B. I phase 통합

```
[ 기존 ]                          [ 통합 후 ]
─────────                         ─────────
solution.py만                     solution.py
                                  + test_subtask1.py
                                  + test_subtask2.py
                                  + test_subtask3.py

I가 코드와 테스트                 I가 코드와 테스트
                                  P의 acceptance 따라 작성
```

### C. V phase 이중 검증 (lite)

```
[ 평소 (저비용) ]
  V → ① I의 sub-task 테스트 실행
       ② Rubric 채점 (기존)
       
[ Stagnation 시 (위기) ]
  V → ① I의 sub-task 테스트 실행
       ② V가 적대적 테스트 추가 작성
       ③ Rubric 채점

조건: 같은 점수 N+1 cycle 반복 시
근거: Phase 1의 stagnation_detector와 자연 결합
```

### D. J phase 강화 — 보상 해킹 감사

```
┌────────────────────────────────────────────┐
│  Audit 체크리스트                              │
│                                              │
│  1. test pass 100% + rubric 0.6              │
│     → 테스트 약함 의심                       │
│                                              │
│  2. assertion 밀도                           │
│     • 1~2줄짜리 = 과도하게 약함              │
│                                              │
│  3. test_hint 커버리지                        │
│     • P가 명시한 케이스 미커버                │
│                                              │
│  4. Sub-task 간 일관성                       │
│     • 모두 trivial 통과 → 구조적 회의         │
│                                              │
│  의심 시 → action="redo_P"                    │
│  reason="test_quality_concern"               │
└────────────────────────────────────────────┘
```

## 4. 4중 안전망 (오라클 약점 방어)

```
┌────────────────────────────────────────────┐
│  ① P: acceptance/test_hint 명시              │
│      I의 테스트가 따라야 할 기준 강제         │
│                                              │
│  ② I: sub-task별 코드 + 테스트                │
│      단, P의 acceptance 참조                  │
│                                              │
│  ③ V: 이중 검증                              │
│      • I 테스트 실행                         │
│      • Rubric 별도 채점                      │
│                                              │
│  ④ J: 사후 감사                              │
│      • test/rubric 격차 감지                 │
│      • assertion 빈약 감지                    │
└────────────────────────────────────────────┘
```
