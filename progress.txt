## Codebase Patterns (최상단 고정)
- v0.2 라이브 검증 통과 — cursor 5 phase, n_queens 1 cycle, weighted_score=1.000 (perf 1.046s/threshold 1.5s).
- 이 프로젝트는 standalone CLI (Python). Claude Code 의존성 없음.
- 모델 추상화: litellm + cursor-agent CLI (subprocess) 듀얼 provider.
  `cursor` / `cursor/<m>` 모델 id는 `_call_cursor_cli`로 분기, 그 외는 litellm.
- cursor-agent는 자체 agentic CLI — 호출 1회에 30s~5min 가능. 모든 phase에 workspace
  넘기면 동일 디렉토리에서 작업 (각 phase가 다른 phase의 산출물을 볼 수 있음).
- cursor-agent는 `~/.cursor/projects/<sanitized-cwd>/` 디렉토리를 작성해야 함 —
  KISTI inode quota 가득 차면 errno 122. `~/.cache/claude-cli-nodejs/-scratch-*` 정리로 해결.
- 오케스트레이션: Code (Python) + LLM Judge 하이브리드. 완전 LLM orchestrator 아님.
- 핵심 사이클: R → P → I → V → J (5단계, plugin과 동일 시맨틱).
- Stateless 원칙: Phase Worker는 fresh context, 모든 상태는 .agent_loop/ 파일에.
- 병렬 처리: v0.1엔 직렬만. v0.3에서 multi-strategy/multi-judge.
- v0.1 미니멀 4 레이어: CLI → Orchestrator → Phase Workers → State.
- 6개 엔진 분리 (Context/Verify/Judge Engine 등)는 v0.2+에서 필요시 추출. YAGNI.
- 한국어 응답 (기술 용어는 영어). 부모는 직접 코드 작성 X, 워커에 위임.
- 빌드 시스템: hatchling (단순함). `[tool.hatch.build.targets.wheel] packages = ["src/agent_loop"]`.
- toml 로딩: Python 3.11+ stdlib `tomllib`, 3.10은 `tomli` 폴백 (`sys.version_info` 분기).
- KISTI에서 검증: `module load python/3.12.4` (실제로 3.14.2가 로드됨), `pip install -e ".[dev]" --user`.
- 모든 phase worker 호출은 `agent_loop.models.call_model(phase, prompt, system, config)`로 단일화.
  - phase는 `Literal["research","plan","implement","verify","judge"]` 5개 고정.
- TaskDir 디렉토리 layout 변경 시 `init()`과 `list_tasks()` 휴리스틱 둘 다 갱신해야 함.
- Test에서 litellm 호출은 monkeypatch로 `litellm.completion` + `litellm.completion_cost` 둘 다 패치.
- v0.1.0 호출 entry point: `python3 -m agent_loop.cli` (KISTI inode 이슈 시). `module load python/3.12.4`는 같은 셸에서만 적용 — 매 Bash invocation 앞에 붙일 것.
- `bench` 명령은 yaml `budget.max_cycles` / `max_redo`가 default; CLI `--cycles` / `--max-redo` 플래그가 override; `--dry-run`이면 task.md만 쓰고 LLM 호출 X (env에 API 키 없을 때 smoke 검증 경로).
- v0.2 Context Engine: `src/agent_loop/context.py` 의 `ContextEngine` 한 클래스가 진입점.
  3-tier memory = `memory/history.jsonl` (audit) + `memory/episodic.md` (Compactor 산출, prompt 노출) +
  `memory/core_facts.md` (CORE: 라인 누적 + v0.1 마이그레이션). 각 phase worker 는 매 호출마다
  `ContextEngine(task_dir).snapshot()` 로 prompt 의 `{memory}` 슬롯을 채우고 끝에
  `append_history({...})` 한 줄을 audit 에 남긴다 (stateless).
- Orchestrator 는 cycle 끝 (judge 직후) 에 `context.compact()` + `context.sensors()` 호출 →
  `metrics.jsonl` 에 `phase: "_cycle_quality"` 행이 추가됨. compact 는 rule-based, LLM 호출 0.
- v0.1 → v0.2 마이그레이션: `memory.txt` 가 비어 있지 않고 `core_facts.md` 가 비어 있으면 1 회만
  복사 + `memory.txt.v0_1.bak` 백업. `init()` 은 idempotent (resume 안전).
- v0.2 Verify Engine: `src/agent_loop/verify_engine.py` 의 `VerifyEngine.evaluate(rubric)` 진입점.
  evaluators/{pytest_runner, benchmark, ast_grep, llm_rubric}.py 4종. `task_dir.artifact_path("rubric.json")`
  존재 시 `workers.run_verify` 가 자동 분기 (legacy LLM 경로 = `_run_verify_llm_legacy` 보존).
  `verify_types.py` 의 `AxisScore` / `VerifyResult` 는 dataclass — circular import 회피용.
- `cli.bench` 가 yaml `success_criteria` → `yaml_to_rubric` → `artifacts/rubric.json` 자동 작성.
- ast_grep 미니 DSL: `` `tok`_count<=N `` / `` `tok` not_in `` / `` `tok` in `` 만 인식. binary_search.yaml 의
  rule 도 DSL 형식으로 변환됨.
- `solution.json` 새 스키마: `{"weighted_score", "summary", "axes": [...]}`. evidence 는 summary 와 동일
  (legacy 호환). v0.1 `{"score": ...}` 도 judge first-cycle 에서 `weighted_score | score` 폴백 처리.
