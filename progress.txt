## Codebase Patterns (최상단 고정)
- 정량 비교는 `benchmarks/compare.py` (baseline=cursor 1-shot vs agent-loop 5-phase, 동일 VerifyEngine 채점). `benchmarks/baseline_runner.py` 가 `cursor-agent --print --output-format text --force --trust --workspace=<ws> "<prompt>"` 1회 호출 후 fenced ```python``` 추출. compare.py 는 `--loop-config` 로 cursor 모든 phase 강제해서 apples-to-apples. 결과 CSV `/tmp/al_compare.csv`, summary JSON `/tmp/al_compare.summary.json`. v0.5.1 1-cycle 기준: n_queens 만 agent-loop 가 baseline 대비 +0.12 점 우위 (perf gate 1.5s 통과 vs baseline 0.76 부분점수); binary_search/palindrome/sort_tuning 는 동점 (1-cycle 한계 — 더 어려운 task 는 cycles>=2 + redo 필요). agent-loop 시간 비용 baseline 의 ~4 배 (70s vs 15s).
- v0.2 라이브 검증 통과 — cursor 5 phase, n_queens 1 cycle, weighted_score=1.000 (perf 1.046s/threshold 1.5s).
- 진짜 cross-vendor verify 증명 (2026-04-27): free-form `agent-loop run` (rubric 없음) + cursor→gemini-2.5-flash 조합으로 verify가 실제 LLM 호출됨 (15.25s, 한국어 evidence). 단 judge는 first cycle 에서 자동 skip ("no prior best") — 다중 vendor judge 증명은 score<1.0 task 필요. claude/default 는 free-form verify 에서 600s timeout 재현됨.
- v0.3 multi-judge + multi-strategy + judge_always_llm 라이브 cross-vendor 검증 통과 (2026-04-27, task_id=9380cc): 3 strategies(cursor/gemini/claude) + 3 judges(동일) 1 cycle 동시 실행 — proposals.json 3개(cursor 1465ch / gemini 2232ch / claude timeout), plan_selector.json `selector_method="heuristic+llm"` winner=gemini-flash (LLM rationale="claude has no usable plan"), plan.md == winner.text byte-identical. judge_result.json `n_judges=3`, `votes_action={"stop":2.0}`, `votes_better={"true":2.0}`, individual 3개(cursor 13s/gemini 34s OK, claude 600s timeout), `fallback=false`. judge_always_llm=true 효과 증명: best=null 인 cycle 1 에서도 cursor/gemini 둘 다 latency_s>0 + reason 텍스트 = 진짜 LLM call. metrics.jsonl plan/judge row 에 `n_strategies`, `n_judges`, `winner_provider`, `consensus_fallback` 모두 정확 기록. 발견 한계: `claude --print` 는 plan/judge prompt 도 600s timeout 재현 (verify 만 아님) — multi-vendor 합의 시 claude leg 은 surviving 2 vendor 로 graceful degrade.
- ⚠️ TOML 작성시 `[[strategies]]`/`[[judges]]` 가 아니라 `[[runtime.strategies]]`/`[[runtime.judges]]`. `judge_always_llm`/`cli_timeout*` 은 `[runtime]` 섹션 안. config.py 의 Runtime 모델이 nested. (라이브 검증 첫 시도 실패 원인.)
- 이 프로젝트는 standalone CLI (Python). Claude Code 의존성 없음.
- 모델 추상화: litellm + 3개 CLI provider (cursor / claude / gemini, 모두 subprocess).
  `cursor/<m>` / `claude/<m>` / `gemini/<m>` 모델 id는 `_cli_provider()` 헬퍼가
  ("cursor"|"claude"|"gemini"|None) 반환 후 분기, None 이면 litellm. cli_timeout
  파라미터로 모든 CLI subprocess 의 timeout 통일 (cursor_timeout 은 호환 alias).
- claude (Claude Code CLI): `--print --output-format text --dangerously-skip-permissions
  --add-dir <ws> "<prompt>"`. `--bare` 는 ANTHROPIC_API_KEY 강제이므로 사용 X (사용자
  OAuth 로그인 유지). 구독 사용자 인증.
- gemini CLI: `-p "<prompt>" -m <model> --output-format text --yolo --skip-trust
  --include-directories <ws>`. Node v22 필수. gemini-2.5-pro 첫 호출 cold start 가
  60+s (90s timeout 부족), gemini-2.5-flash 는 ~8s. judge phase 에 flash 권장.
- cursor-agent는 자체 agentic CLI — 호출 1회에 30s~5min 가능. 모든 phase에 workspace
  넘기면 동일 디렉토리에서 작업 (각 phase가 다른 phase의 산출물을 볼 수 있음).
- cursor-agent는 `~/.cursor/projects/<sanitized-cwd>/` 디렉토리를 작성해야 함 —
  KISTI inode quota 가득 차면 errno 122. `~/.cache/claude-cli-nodejs/-scratch-*` 정리로 해결.
- 오케스트레이션: Code (Python) + LLM Judge 하이브리드. 완전 LLM orchestrator 아님.
- 핵심 사이클: R → P → I → V → J (5단계, plugin과 동일 시맨틱).
- Stateless 원칙: Phase Worker는 fresh context, 모든 상태는 .agent_loop/ 파일에.
- 병렬 처리: v0.1엔 직렬만. v0.3에서 multi-strategy/multi-judge.
- v0.1 미니멀 4 레이어: CLI → Orchestrator → Phase Workers → State.
- 6개 엔진 분리 (Context/Verify/Judge Engine 등)는 v0.2+에서 필요시 추출. YAGNI.
- 한국어 응답 (기술 용어는 영어). 부모는 직접 코드 작성 X, 워커에 위임.
- 빌드 시스템: hatchling (단순함). `[tool.hatch.build.targets.wheel] packages = ["src/agent_loop"]`.
- toml 로딩: Python 3.11+ stdlib `tomllib`, 3.10은 `tomli` 폴백 (`sys.version_info` 분기).
- KISTI에서 검증: `module load python/3.12.4` (실제로 3.14.2가 로드됨), `pip install -e ".[dev]" --user`.
- 모든 phase worker 호출은 `agent_loop.models.call_model(phase, prompt, system, config)`로 단일화.
  - phase는 `Literal["research","plan","implement","verify","judge"]` 5개 고정.
- TaskDir 디렉토리 layout 변경 시 `init()`과 `list_tasks()` 휴리스틱 둘 다 갱신해야 함.
- Test에서 litellm 호출은 monkeypatch로 `litellm.completion` + `litellm.completion_cost` 둘 다 패치.
- v0.1.0 호출 entry point: `python3 -m agent_loop.cli` (KISTI inode 이슈 시). `module load python/3.12.4`는 같은 셸에서만 적용 — 매 Bash invocation 앞에 붙일 것.
- `bench` 명령은 yaml `budget.max_cycles` / `max_redo`가 default; CLI `--cycles` / `--max-redo` 플래그가 override; `--dry-run`이면 task.md만 쓰고 LLM 호출 X (env에 API 키 없을 때 smoke 검증 경로).
- v0.2 Context Engine: `src/agent_loop/context.py` 의 `ContextEngine` 한 클래스가 진입점.
  3-tier memory = `memory/history.jsonl` (audit) + `memory/episodic.md` (Compactor 산출, prompt 노출) +
  `memory/core_facts.md` (CORE: 라인 누적 + v0.1 마이그레이션). 각 phase worker 는 매 호출마다
  `ContextEngine(task_dir).snapshot()` 로 prompt 의 `{memory}` 슬롯을 채우고 끝에
  `append_history({...})` 한 줄을 audit 에 남긴다 (stateless).
- Orchestrator 는 cycle 끝 (judge 직후) 에 `context.compact()` + `context.sensors()` 호출 →
  `metrics.jsonl` 에 `phase: "_cycle_quality"` 행이 추가됨. compact 는 rule-based, LLM 호출 0.
- v0.1 → v0.2 마이그레이션: `memory.txt` 가 비어 있지 않고 `core_facts.md` 가 비어 있으면 1 회만
  복사 + `memory.txt.v0_1.bak` 백업. `init()` 은 idempotent (resume 안전).
- v0.2 Verify Engine: `src/agent_loop/verify_engine.py` 의 `VerifyEngine.evaluate(rubric)` 진입점.
  evaluators/{pytest_runner, benchmark, ast_grep, llm_rubric}.py 4종. `task_dir.artifact_path("rubric.json")`
  존재 시 `workers.run_verify` 가 자동 분기 (legacy LLM 경로 = `_run_verify_llm_legacy` 보존).
  `verify_types.py` 의 `AxisScore` / `VerifyResult` 는 dataclass — circular import 회피용.
- `cli.bench` 가 yaml `success_criteria` → `yaml_to_rubric` → `artifacts/rubric.json` 자동 작성.
- ast_grep 미니 DSL: `` `tok`_count<=N `` / `` `tok` not_in `` / `` `tok` in `` 만 인식. binary_search.yaml 의
  rule 도 DSL 형식으로 변환됨.
- `solution.json` 새 스키마: `{"weighted_score", "summary", "axes": [...]}`. evidence 는 summary 와 동일
  (legacy 호환). v0.1 `{"score": ...}` 도 judge first-cycle 에서 `weighted_score | score` 폴백 처리.
- v0.3 Multi-judge: `src/agent_loop/judge_engine.py` 의 `JudgeEngine.consensus(judges, prompt)` 진입점.
  ThreadPoolExecutor 로 N provider 병렬 호출 + weighted majority 합의. `Runtime.judges` (list[JudgeSpec])
  가 비어있지 않으면 `workers.run_judge` 가 `_run_judge_multi` 분기 (first-cycle short-circuit 보존).
  `judge_result.json` 에 `consensus` 키 추가 (single 모드는 변경 X — backward compat). 합의 알고리즘:
  action 동률시 'stop' 우선 → 알파벳 첫번째, better 동률시 False (보수). all-fail 시 single fallback +
  `consensus.fallback=True`. CLI: `--judge` 반복 가능 플래그, ENV `AGENT_LOOP_RUNTIME_JUDGES` 콤마구분.
- v0.3 Multi-strategy: `src/agent_loop/strategy_engine.py` 의 `StrategyEngine.fanout(strategies, prompt)`
  진입점. ThreadPoolExecutor 로 N provider 병렬 plan 호출 + Selector(휴리스틱+LLM) winner 1개. 휴리스틱:
  length/fenced/steps/headers 가중합. LLM rubric 은 `cfg.models.plan` 1회 호출 (실패시 structural fallback,
  selector_method="fallback"). `Runtime.strategies` (list[StrategySpec]) 가 비어있지 않으면
  `workers.run_plan` 이 `_run_plan_multi` 분기. winner.text 그대로 `plan.md` 에 저장 (downstream 호환),
  전체 audit 은 `proposals.json`, selector 결과는 `plan_selector.json`. 단일 proposal 시 selector 호출 X.
  All-fail → `AllStrategiesFailed` raise (single fallback 안 함, 명시적 에러). CLI: `--strategy` 반복가능,
  ENV `AGENT_LOOP_RUNTIME_STRATEGIES`. JudgeEngine 의 `_call_one` 패턴(Config 복제 + model 덮어쓰기)을
  strategy_engine 안에 복제 — 두 엔진 간 import X (engine 결합 회피).
- v0.3.1 cli_timeout phase-aware: `Runtime.cli_timeout` (default 600) + 5개 per-phase
  override 필드 (`cli_timeout_research|plan|implement|verify|judge`) + 헬퍼
  `cli_timeout_for(phase)`. `models.call_model` 은 `cli_timeout=None` 일 때 `cfg.runtime.
  cli_timeout_for(phase)` 로 폴백 (legacy `cursor_timeout` 은 phase 기본=600 인 경우에만 honor).
  ENV: `AGENT_LOOP_RUNTIME_CLI_TIMEOUT[_<PHASE>]`. CLI: `--cli-timeout` /
  `--cli-timeout-{verify,judge}` (run + bench).
- v0.3.1 judge_always_llm: `Runtime.judge_always_llm` (default False). True 면 first-cycle
  short-circuit 분기 통째 비활성 → single/multi 둘 다 cycle 1 에서도 LLM 호출. multi 모드는
  `best_solution.json` 부재 시 `{}` stub 으로 prompt 렌더. ENV `AGENT_LOOP_RUNTIME_JUDGE_
  ALWAYS_LLM=true`. CLI `--judge-always-llm` flag (run + bench). v0.3 backward compat 보존
  (default off). free-form 1-cycle multi-judge 진짜 검증 의 게이트.
- v0.3.2 claude tool-block: `_call_claude_cli` 는 `--allowedTools=NoneSuch` (phantom 도구
  이름) 로 모든 도구 차단해서 self-invoke timeout 회피해야 함 — verify/judge 긴 prompt 에서
  critical. `--add-dir=<workspace>` 등호 형식 필수 (nargs='*' 인 `--allowedTools` 가
  workspace 를 삼키는 것 방지). 패치 전 free-form claude verify=600s timeout, 패치 후 14.6s.
  claude ping 도 10.3s→7.1s 단축.
- v0.4.0 cross-task memory: `~/.agent-loop/global/patterns.md` (CORE: 라인 dedup 누적) +
  `task_index.jsonl` (한 줄 요약). `ContextEngine(td, global_root=..., cross_task=True,
  global_max_chars=4000)` 로 생성. `commit_to_global({task_id, weighted_score, cycles,
  task_md_first_line, final_status})` 가 orchestrator `_finalize` 에서 모든 exit path
  마다 호출됨 (try/except 로 격리). `MemorySnapshot.global_patterns` 가 `# Global Patterns
  (cross-task)` 섹션으로 prompt {memory} 슬롯에 자동 합쳐짐. cross_task=False 면 글로벌
  IO 0. dedup 은 exact match. CLI: `agent-loop memory {show,list,wipe,path}`.
  ENV: `AGENT_LOOP_RUNTIME_CROSS_TASK_MEMORY[_DIR|_MAX_CHARS]`. flag `--no-cross-task`
  (한 번 끄기). MCP server 는 v0.4.1 에 `commit_to_global`/`_load_global_patterns` hooks
  재사용. Privacy: 코드/prompt 본문 글로벌 X. task_md_first_line 만 (사용자가 첫 줄에
  민감 정보 X 가정).
- v0.4.1 auto-rubric: `src/agent_loop/auto_rubric.py` 의 `generate_rubric(task, findings,
  cfg)` 가 R phase 끝에서 별도 `call_model("research", ...)` 1회 호출 → `rubric_auto.json`
  부산물. `workers.run_verify` 우선순위: `rubric.json` > `rubric_auto.json` > legacy LLM
  (auto_rubric=False 면 auto_rubric 무시). 모든 axis 는 `evaluator: "llm_rubric"` 강제
  (LLM 이 task 만 보고 작성하므로 pytest/benchmark 코드 못 만듦; v0.4.2 후보). axes 최소
  2 + weights 합 1.0 normalize, 단일 axis 또는 weight<=0 거부. error isolation: 파싱/검증
  실패 시 `findings.md` 는 살리고 verify 만 legacy fallback (warn-and-skip). Config:
  `Runtime.auto_rubric: bool = True` (default ON). ENV `AGENT_LOOP_RUNTIME_AUTO_RUBRIC=
  false`. CLI `--no-auto-rubric` (run only — bench 는 yaml 우선이라 무관).
- auto-rubric 은 R 후 별도 call_model — 우선순위 `rubric.json > rubric_auto.json > legacy`.
- v0.4.2 privacy fix: `workers._memory_text(task_dir, config)` 시그니처 — config 로
  `runtime.cross_task_memory*` 를 `ContextEngine` 에 전달해야 worker prompt 가
  `--no-cross-task` 플래그를 honor 함. 5개 phase worker (R/P/I/V/J) 호출처 모두 갱신.
  (이전엔 `ContextEngine(task_dir)` default 로 만들어서 config 가 무시됨.)
- v0.4.2 metric fix: `auto_rubric.generate_rubric` → `RubricGeneration(rubric, response)`
  반환. `_maybe_generate_auto_rubric` 가 `task_dir.append_metric({"phase":"_auto_rubric",
  ...cost_usd/tokens_in/tokens_out/latency_s})` 한 row 추가 + `run_research` 가 ModelResponse
  의 cost/latency/tokens 합산해서 반환 → 오케스트레이터 budget guard 가 auto-rubric 비용
  포함. underscore prefix(`_auto_rubric`) 는 orchestrator hardcode phase set 회피
  (v0.2 `_cycle_quality` 와 동일 패턴).
- v0.5.0 MCP server: `src/agent_loop/mcp/` (server.py / handlers.py / protocol.py /
  __init__.py) 신규. JSON-RPC 2.0 over stdio (HTTP 는 v0.5.x 후보), 의존성 0 (stdlib
  only — `mcp` SDK 도입 X). 6 tools (`agent_loop.{run,list,status,resume,bench,
  memory_show}`) + 4 resources (`agent-loop://task/{id}/{solution,memory,metrics}`,
  `agent-loop://global/patterns`). Privacy: `cross_task_memory=False` 면 global
  resource 는 `ERR_PRIVACY_DISABLED (-32000)` 반환, 다른 task 의 task.md / prompt 는
  resource URI scheme 자체에 없음. server 는 long-running 이지만 매 tool call 마다
  `Orchestrator(td, cfg)` 새로 만듦 (stateless 보존). CLI: `agent-loop mcp serve` /
  `agent-loop mcp tools` / `agent-loop mcp resources`. handler 분기: `Handlers.dispatch
  (method, params, rid) -> Response` 가 `_PrivacyError`/`_NotFoundError`/`_BadParams
  Error`/일반 Exception 모두 잡아 JSON-RPC error 로 wrap (server loop 절대 안 죽음).
- v0.5.0 KNOWN BUG → v0.5.1 FIXED: orchestrator 의 rich Console (default
  `Console()`) 가 stdout 으로 progress 를 찍어 MCP stdio JSON-RPC frame 과
  interleave 했음. v0.5.1 에서 `Handlers._tool_run/_resume/_bench` 가

- v0.5.1 stdout/stderr 분리: `Orchestrator(td, cfg, console=...)` 가 진실의 단일
  소스. CLI (`run`/`bench`/`resume`) 는 stdout `Console()` (사용자가 progress 봄),
  MCP `_tool_run`/`_tool_resume`/`_tool_bench` 는 `_stderr_console()` 헬퍼로
  `Console(file=sys.stderr, force_terminal=False)` 주입 → stdout 은 JSON-RPC

  안에서 호출되면 반드시 stderr console 주입 필수.

  tools_list/2 tools_call/resources_list/3 errors/exit), `tests/e2e_mcp_live_run.py`
